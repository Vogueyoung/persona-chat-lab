# Prompt versions — v1 → v2

This document records why v2 exists and what it changes. The raw linter
output is in `docs/transcripts/prompt-scores.md` (auto-generated by
`scripts/eval_prompts.py`).

## Headline numbers

| Version | Score (mean per prompt) | % of 18 |
|---------|------------------------:|--------:|
| `v1` | 12.0 / 18 | 66.7% |
| `v2` | 18.0 / 18 | 100.0% |

**Delta:** +6.0 points per prompt, +33.3 percentage points on the
12-criterion structural rubric (see `evals/rubrics.md`).

The numbers above come from the structural linter (key-less). The
LLM-judge harness (`scripts/run_eval.py`) is wired and ready to run —
it produces a comparable 5-dimension score sheet when
`OPENAI_API_KEY` is set, at a cost of roughly $0.20–$0.50 per full
20-prompt × 2-version run.

## Where v2 beats v1 (per-criterion deltas)

All criteria that v2 improves are **structural** choices in the
renderer, not content rewrites. Specifically:

| Criterion | v1 | v2 | Why v2 is better |
|---|---:|---:|---|
| Few-shot primacy — examples placed above personality description | 0 / 6 | **6 / 6** | Primacy effect: the model anchors tone from concrete examples before encountering abstract rules. v1 put few-shot last, which is backwards for persona stickiness. |
| Guardrails separated — identity protection split from topic safety | 0 / 6 | **6 / 6** | In v1, "don't reveal you're an AI" sat in the same section as "avoid politics". Merging dilutes both. Splitting lets identity protection get its own stronger wording ("AI·언어모델·프로그램·봇·GPT·챗봇이 아닙니다. 절대 그렇게 밝히지 않습니다"). |
| Real-world question handling | 0 / 6 | **6 / 6** | v1 had no guidance for "오늘 서울 날씨 어때?" style probes. The model would sometimes break character to answer. v2 adds an explicit section: stay in-world, reinterpret or deny. |

## What v2 does **not** change

Keeping these constant makes the delta honest:

- Character card schema — same YAML contract
- Tone phrasing table (`_TONE_PHRASES`) — same natural-language directives
- RAG block wording — same "자료에 없는 내용 지어내지 않는다" policy
- Character count, character identities, backstories, forbidden topics

If v2 wins on the LLM-judge rubric too (when run with a key), we can
attribute the delta specifically to structural changes, not to content
rewrites in the character cards.

## Expected v2 wins on the LLM-judge rubric

These are hypotheses — the linter can't measure them directly. They
are what v2 is designed to improve:

- **persona_consistency** — few-shot-first should tighten tone match,
  especially for characters with strong voice (Aria's 본인/그대, Zen's
  terse ...어).
- **safety / break-character** — separated identity guardrail with
  stronger wording should resist "너 AI지?" / "OOC" prompts more
  reliably. Specifically tested by `break-01` through `break-04` in the
  dataset.
- **real-world probes** — explicit handling section should stop
  responses to `real-01`..`real-03` from breaking character.

Expected non-wins or small deltas:

- **length_appropriateness** — structural changes shouldn't affect
  length much; both versions carry the same verbosity directive.
- **factual_grounding** — RAG wording is identical; any delta here
  would be a confound.

## How to reproduce

```powershell
# Structural (no key)
uv run python -m scripts.eval_prompts --write docs/transcripts/prompt-scores.md

# LLM-judge (key required)
uv run python -m scripts.run_eval --versions v1,v2 --write docs/transcripts/llm-judge.md
```

## What v3 would look like

v3 is not in scope for the 4-day portfolio, but the natural next
improvement is **dynamic few-shot selection**: instead of always
showing the 3 canned examples from the character card, pick the 2–3
examples most similar to the user's current message (via the same
embedding index as lore RAG). That addresses the one v1/v2 shared
weakness: the examples aren't sensitive to the query.
